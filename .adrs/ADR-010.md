# ADR-010: bincode + Zstd over FlatBuffers for Wire Serialization

> **This ADR has been superseded. See [ADR-001](./ADR-001.md) for the current decision.**

**Status:** Superseded by ADR-001
**Date:** 2026-05-12
**Related ADRs:** ADR-001
**Standards:** N/A

## Context

The original requirements specified FlatBuffers for zero-copy deserialization of patch data. As of v5.4.0, the wire protocol uses bincode (compact binary serialization) with Zstd compression.

## Decision

**Use bincode + Zstd for wire serialization. Do not migrate to FlatBuffers.**

## Rationale

1. **Serde ecosystem integration.** bincode uses standard Rust `#[derive(Serialize, Deserialize)]` annotations. Every data type in the codebase (Patch, Blob, Commit, Branch) already derives serde for JSON debug output, making bincode a drop-in replacement. FlatBuffers requires a separate schema language and code generation step.

2. **Zstd compression dominates wire size.** After Zstd compression, the serialization format's wire size differences are negligible. For Patch data (JSON/YAML/XML diffs), Zstd achieves 10-20:1 compression ratios. The choice of pre-compression serialization format is a rounding error.

3. **Patch payloads are not large enough for zero-copy.** Patches are typically 1-50 KB (pre-compression). Zero-copy deserialization benefits are measurable at MB+ payloads. The allocation overhead of bincode deserialization for KB-sized payloads is immeasurable.

4. **Build complexity.** FlatBuffers requires `flatc` in the build pipeline, schema maintenance, and generated code management. bincode requires none of these.

5. **Forward/backward compatibility.** bincode's field numbering approach handles schema evolution. Adding fields is backward-compatible. Removing fields requires explicit handling but is straightforward.

## Consequences

- **Positive:** No build dependencies. Standard serde derives. No schema language to learn. Full Rust type safety without code generation.
- **Negative:** No zero-copy deserialization for large payloads. No cross-language schema for non-Rust consumers. Binary format is Rust-specific (not a standard like ProtoBuf/FlatBuffers).
- **Acceptable risk:** Current consumers are all Rust (CLI, Daemon, Hub). Non-Rust language bindings (Node.js, Python) use a different serialization path (JSON over napi-rs/PyO3). Cross-language wire compatibility is not required.

## Alternatives Considered

| Alternative | Rejected Because |
|-------------|-----------------|
| FlatBuffers | Build complexity, schema language overhead, zero-copy benefit marginal for patch-sized payloads |
| Protocol Buffers | Build complexity (protoc), no zero-copy, Google dependency |
| Cap'n Proto | Build complexity, less mature Rust ecosystem than serde |
| MessagePack | Similar to bincode but larger wire size; no compelling advantage |
| JSON | 2-3x larger wire size before compression; no type safety on wire |
| CBOR | No serde ecosystem advantage over bincode; JSON compatibility irrelevant |

## Future Migration Path

If cross-language wire compatibility becomes necessary (e.g., Go/Python Hub clients consuming wire format directly):
1. Define a `.proto` schema for the core wire types.
2. Implement `Serialize`/`Deserialize` via ProtoBuf alongside bincode.
3. Negotiate wire format during V2 handshake.
4. Support both formats for backward compatibility.
