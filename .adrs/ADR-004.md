# ADR-004: Hub Database

## Status
Accepted

## Context
Suture Hub stores project metadata, user permissions, patch-DAG commit history, and liveness state. The original spec called for PostgreSQL for distributed relational storage. PostgreSQL provides robustness for high-concurrency writes, replication, and complex queries. However, Suture's current deployment model is single-node hubs and small clusters (2-3 nodes), where SQLite with WAL mode provides sufficient concurrency and dramatically simpler operations.

## Decision
Use SQLite with WAL (Write-Ahead Logging) mode via `rusqlite`. WAL mode allows concurrent reads during writes and provides crash recovery without a separate write-ahead log file. A single SQLite database file stores all hub state.

## Consequences
### Positive
- Zero configuration: no database server to install, configure, or monitor
- Single-file deployment simplifies backup, migration, and disaster recovery (copy the file)
- WAL mode supports concurrent reads and serialized writes; sufficient for hub-scale workloads (tens of concurrent users)
- `busy_timeout` configuration handles write contention gracefully
- Eliminates an entire operational dependency (PostgreSQL connection pooling, replication, vacuuming)

### Negative
- No built-in replication; multi-hub setups require application-level sync
- Write throughput is limited by single-writer model; high-concurrency write workloads will hit `SQLITE_BUSY`
- No advanced SQL features: no CTE materialization hints, no partitioned tables, no row-level security
- Maximum practical database size is ~140 TB (SQLite limit) but performance degrades above ~50 GB

### Risks
- If a customer requires multi-region active-active replication, SQLite will not suffice. Mitigation: SQLite + Raft (via `raft-rs`) can provide consensus-based replication for multi-hub deployments without introducing PostgreSQL.
- High-concurrency write patterns (many users committing simultaneously) may cause contention. Mitigation: increase `busy_timeout`, batch writes in the daemon, and profile at scale.

## Alternatives Considered
| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|--------------|
| PostgreSQL | Mature replication (streaming, logical), high write throughput, advanced SQL, row-level security | Requires a database server process, connection pooling (PgBouncer), operational overhead, Docker dependency | Operational complexity not justified for current deployment scale |
| MySQL/MariaDB | Widely deployed, replication support | Similar operational overhead to PostgreSQL, weaker SQL standard compliance | No advantage over PostgreSQL if migration were needed |
| CockroachDB | Distributed SQL, automatic sharding, PostgreSQL wire compatibility | Significant operational complexity, requires multi-node cluster | Overkill for single-node and small cluster deployments |
| sled (embedded Rust KV store) | Fast, no external dependency, embedded | Not relational (no SQL), limited query capability, immature compared to SQLite | Suture's query patterns (joins, aggregations on metadata) require relational access |
