# ADR-002: Transport Protocol

## Status
Accepted

## Context
Suture Hub communicates with daemon clients over the network for sync, lock acquisition, and patch exchange. The original spec called for QUIC (via `quinn`) to avoid TCP head-of-line blocking and improve performance on lossy networks. QUIC provides 0-RTT connection establishment and connection migration, which are valuable for mobile or roaming clients. However, Suture's primary traffic pattern is daemon-to-hub sync from fixed workstations on local networks or datacenter environments, where packet loss is negligible.

## Decision
Use TCP as the transport layer with `tonic` (gRPC) for the RPC framework. Zstd delta-compression is applied at the application layer to reduce bandwidth usage. TLS 1.3 termination handles encryption.

## Consequences
### Positive
- tonic/gRPC over TCP is mature and well-documented; all 14 RPCs implemented and stable
- TCP provides reliable ordered delivery; no need for application-level reassembly
- Zstd delta-compression reduces effective bandwidth by 3-5x for patch payloads, compensating for TCP's HOL blocking on rare loss events
- No need to implement connection migration logic; daemon connections are long-lived and stationary
- Standard tooling for observability (grpcurl, Envoy proxies, Linkerd)

### Negative
- TCP HOL blocking means a single lost packet stalls all streams on that connection until retransmission (~1 RTT on LAN)
- No 0-RTT connection establishment; TLS handshake adds ~1 RTT on reconnect
- Cannot take advantage of UDP-based congestion control innovations

### Risks
- Remote editors on high-latency, lossy links (e.g., satellite, cellular) may experience degraded sync performance. Mitigation: QUIC can be introduced as an optional transport behind a feature flag if demand materializes.

## Alternatives Considered
| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|--------------|
| QUIC via quinn | No HOL blocking, 0-RTT, connection migration, UDP-based | Less mature gRPC interop (grpc-go/grpc-rust QUIC support is limited), complex state machine, TLS 1.3 mandatory (no cleartext for debugging) | Current deployment is LAN/datacenter where TCP HOL blocking is negligible |
| HTTP/2 without gRPC | Simpler than gRPC, streaming support | No code generation, manual routing, no strongly-typed client stubs | gRPC provides significant developer ergonomics for 14 RPCs |
| Raw TCP with custom framing | Maximum control, minimal overhead | Reinventing multiplexing, flow control, and keepalive; no ecosystem tooling | Unnecessary engineering for a problem gRPC solves well |
