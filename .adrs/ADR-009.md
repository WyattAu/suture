# ADR-009: TCP + Zstd over QUIC for Hub Transport

**Status:** Accepted
**Date:** 2026-05-12
**Related ADRs:** None
**Standards:** N/A

## Context

The original requirements specified gRPC over QUIC using the `quinn` crate for encrypted, multiplexed, low-latency communication. As of v5.3.1, the Hub uses TCP (tonic/HTTP2) with Zstd delta compression.

## Decision

**Use TCP with tonic/gRPC and Zstd compression. Do not implement QUIC transport.**

## Rationale

1. **Adequate latency profile.** VCS operations (push, pull, clone, fetch) are bulk data transfers, not latency-sensitive streaming. The dominant latency component is network RTT, not transport-layer overhead. TCP + HTTP2 multiplexing is sufficient.

2. **Zstd delta compression.** The wire protocol compresses patch deltas with Zstd before transmission. For repetitive structured data (JSON/YAML/XML patches), compression ratios regularly exceed 10:1, making wire overhead negligible relative to payload size.

3. **Implementation maturity.** tonic/h2 is the most mature Rust gRPC implementation with 30M+ downloads. quinn is newer and has fewer production deployments. HTTP2 has better observability tooling (load balancers, proxies, Wireshark decoders).

4. **Operational compatibility.** TCP is universally routable. QUIC (UDP) is sometimes blocked by corporate firewalls and cloud network policies. TCP ensures maximum deployment compatibility.

5. **Complexity budget.** QUIC adds implementation complexity: connection migration, 0-RTT handshake, flow control distinct from HTTP2 flow control. None of these features provide clear benefit for bulk data transfer patterns.

## Consequences

- **Positive:** Stable, well-tested transport. Universal network compatibility. Rich observability ecosystem. Lower implementation complexity.
- **Negative:** No 0-RTT reconnection. Higher latency under packet loss (Head-of-Line blocking in HTTP2 streams). No connection migration across network changes.
- **Acceptable risk:** VCS operations are not interactive (no sub-millisecond latency requirement). Packet loss on modern networks is negligible. Connection migration is irrelevant for server-to-server or client-to-hub patterns.

## Alternatives Considered

| Alternative | Rejected Because |
|-------------|-----------------|
| QUIC (quinn) | Implementation complexity, firewall issues, insufficient benefit for bulk transfer pattern |
| Plain HTTP/1.1 | No multiplexing, no streaming, higher latency |
| WebSocket | Unnecessary bidirectional real-time channel; gRPC unary + server streaming covers all RPCs |
| Custom UDP protocol | Build-vs-buy calculus strongly favors standard protocols |

## Future Migration Path

If future use cases require QUIC (e.g., real-time collaborative editing with sub-100ms latency requirements):
1. Implement `Transport` trait with `TcpTransport` and `QuicTransport` variants.
2. Gate QUIC behind a feature flag (`quic-transport`).
3. Negotiate transport during the V2 handshake.
4. Fall back to TCP if QUIC is unavailable.
