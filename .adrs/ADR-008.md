# ADR-008: SQLite over PostgreSQL for Hub Storage

> **This ADR has been superseded. See [ADR-004](./ADR-004.md) for the current decision.**

**Status:** Superseded by ADR-004
**Date:** 2026-05-12
**Related ADRs:** ADR-004
**Standards:** N/A

## Context

The original requirements specified PostgreSQL as the distributed relational store for the Hub. As of v5.4.0, SQLite with WAL mode serves all Hub storage needs.

## Decision

**Use SQLite as the primary Hub database. Do not migrate to PostgreSQL.**

## Rationale

1. **Operational simplicity.** SQLite requires zero configuration, no separate process, no connection pooling, and no authentication infrastructure. Single-file backup (`cp hub.db backup.db`) is trivial.

2. **Sufficient scale.** The current deployment model targets small-to-medium teams (1-100 users). SQLite comfortably handles:
   - Read throughput: 50K+ queries/sec with WAL mode
   - Write throughput: 10K+ transactions/sec
   - Database size: Up to 281 TB
   - Concurrent readers: Unlimited with WAL mode

3. **Raft integration.** The Raft consensus layer replicates committed state across nodes. Each node maintains its own SQLite database applied deterministically from the Raft log. A shared PostgreSQL instance would add a single point of failure and defeat the purpose of Raft-based replication.

4. **Deployment model.** Self-hosted users run `suture hub` as a single binary. Adding PostgreSQL as a prerequisite would significantly increase the operational burden and reduce adoption.

5. **Transaction integrity.** SQLite's WAL mode provides atomic commit, crash recovery, and serializable isolation — sufficient for all Hub operations.

6. **Data sovereignty.** Air-gapped deployments (requirement §6) are impossible with managed PostgreSQL services and difficult with self-managed PostgreSQL clusters. SQLite enables truly disconnected operation.

## Consequences

- **Positive:** Single-binary deployment. Zero external DB dependencies. Lower operational complexity. Trivial backup/restore.
- **Negative:** Limited horizontal read scaling without read replicas. No native connection pooling across processes. Maximum concurrent writers is 1 (WAL mode limitation).
- **Acceptable risk:** Write throughput is not a bottleneck for VCS metadata operations (primarily reads). If write pressure emerges, it can be amortized through the Raft proposer batching.

## Alternatives Considered

| Alternative | Rejected Because |
|-------------|-----------------|
| PostgreSQL | Operational complexity, multi-process requirement, defeats single-binary deployment story |
| MySQL | Same as PostgreSQL plus weaker SQLite compatibility |
| Redis (for hot path) | Added infrastructure, another stateful service to manage |
| Sqlite + Litestream (S3 replication) | Requires always-running S3 connection, violates air-gapped mode |

## Future Migration Path

If scale requirements exceed SQLite's capabilities, the migration path is:
1. Introduce `HubStorage` trait with `SqliteHubStorage` and `PostgresHubStorage` implementations.
2. Use feature flags (`postgres-backend`) to enable PostgreSQL support.
3. Maintain SQLite as the default for self-hosted and single-node deployments.
4. Document migration procedure: dump SQLite → load PostgreSQL.
