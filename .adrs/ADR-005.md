# ADR-005: Liveness Tracking

## Status
Accepted

## Context
Suture tracks which users are currently active in which sequences/timelines to display presence information and prevent conflicting edits. The original spec called for Redis with a sliding window counter for real-time liveness tracking. Redis provides sub-millisecond reads, pub/sub for instant notifications, and built-in TTL for automatic expiry. However, introducing Redis as a runtime dependency adds operational complexity (a separate server process, memory management, persistence configuration) that is disproportionate to the current scale.

## Decision
Use a SQLite-backed rate limiter for liveness tracking. Client heartbeats are written to the existing SQLite database with TTL-based expiry. Liveness queries read directly from the same database used for project metadata.

## Consequences
### Positive
- No additional runtime dependency; liveness data lives in the existing SQLite database
- Consistent operational model: single database file for all hub state
- TTL-based expiry is handled by periodic cleanup queries, not a separate data structure
- Simplified deployment: no Redis server to install, configure, or monitor

### Negative
- Liveness reads are bounded by SQLite query latency (~100 us for indexed lookups) vs Redis (~10 us for GET)
- No built-in pub/sub mechanism; presence updates require polling or a separate notification channel
- Write amplification: every heartbeat is a SQLite write, which triggers WAL fsync
- Sliding window counters are less efficient in SQL than Redis ZRANGEBYSCORE

### Risks
- If presence needs sub-10ms update latency for real-time collaborative editing (e.g., cursor sharing), SQLite polling may be too slow. Mitigation: add in-process pub/sub (tokio broadcast) for local notifications; SQLite remains the source of truth for persistence.
- High heartbeat frequency from many clients could overwhelm SQLite's single-writer model. Mitigation: batch heartbeat updates and use coarser granularity (2-5s intervals).

## Alternatives Considered
| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|--------------|
| Redis | Sub-ms latency, native pub/sub, atomic TTL expiry, sliding window with sorted sets | Separate server process, memory consumption, persistence configuration, operational complexity | Current user count does not justify the operational overhead |
| etcd | Strongly consistent KV store, watch API for change notifications | Heavier than Redis, designed for configuration not presence, Go-based | Overkill for liveness tracking; adds a third distributed system dependency |
| In-memory HashMap (process-local) | Zero-latency reads, simplest implementation | Lost on restart, not shared across hub instances, no persistence | Presence data must survive hub restarts to provide accurate state |
| SQLite + in-process broadcast | Low latency for local subscribers, SQLite for persistence | Only works within a single process; remote clients must poll | Current architecture is single-process hub; acceptable trade-off |
