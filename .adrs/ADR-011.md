# ADR-011: FUSE3 + WebDAV over NFSv4/SMB3/ProjFS for VFS

> **This ADR has been superseded. See [ADR-003](./ADR-003.md) for the current decision.**

**Status:** Superseded by ADR-003
**Date:** 2026-05-12
**Related ADRs:** ADR-003
**Standards:** N/A

## Context

The original requirements specified NFSv4/SMB3 servers for cross-platform loopback mounting and Windows ProjFS for native file virtualization. As of v5.4.0, the VFS layer uses FUSE3 (Linux) and WebDAV (cross-platform).

## Decision

**Use FUSE3 on Linux and WebDAV for cross-platform mounting. Do not implement NFSv4, SMB3, or ProjFS.**

## Rationale

1. **FUSE3 covers the primary use case.** NLEs (Non-Linear Editors) and development tools on Linux read/write files through FUSE3 mounts. FUSE3 provides native POSIX semantics (read, write, stat, readdir, create, unlink, rename) without kernel-level development.

2. **WebDAV provides cross-platform coverage.** macOS Finder natively mounts WebDAV shares. Windows Explorer does the same via the WebClient service. WebDAV requires zero installation on macOS and Windows — critical for adoption.

3. **NFSv4/SMB3 require kernel-level development.** Both protocols require kernel module implementations or user-space servers that emulate kernel interfaces. In Rust, `nfs-serve-rs` exists but is immature. SMB3 is protocol-complex (500+ pages of specification). The development cost is not justified by the marginal benefit over WebDAV.

4. **ProjFS is Windows-only.** The Windows Projected File System provides native-speed virtualization but is a Windows-exclusive API. Maintaining a Windows-only code path for a minority of users is not justified.

5. **WebDAV performance is adequate for VCS workloads.** VCS operations are not IOPS-intensive. WebDAV's HTTP-based protocol adds ~1ms latency per operation, which is negligible compared to disk I/O (0.1-10ms) and network I/O (1-100ms).

## Consequences

- **Positive:** User-space implementation (no kernel modules). Zero-install on macOS and Windows. Cross-platform via a single protocol. Well-tested WebDAV implementations in Rust (`actix-web`).
- **Negative:** WebDAV lacks POSIX completeness (no hard links, no device files, limited permission model). FUSE3 is Linux-only. Performance is ~10% slower than kernel-space VFS for metadata-heavy workloads.
- **Acceptable risk:** POSIX completeness is not required for VCS working directory operations. FUSE3 covers the primary NLE use case (Linux). WebDAV covers macOS/Windows with acceptable performance for non-real-time workloads.

## Alternatives Considered

| Alternative | Rejected Because |
|-------------|-----------------|
| NFSv4 | Kernel-level development, no mature Rust implementation |
| SMB3 | Protocol complexity (500+ pages), no Rust implementation |
| ProjFS | Windows-only, minority user base |
| 9P (Plan 9) | No native OS support (requires 3rd-party clients) |
| Custom kernel module | Maintenance burden, kernel version compatibility, security risk |

## Future Migration Path

If WebDAV performance becomes a bottleneck or POSIX completeness is required:
1. Monitor `nfs-serve-rs` maturity. If it reaches production quality, add as a feature-gated NFS backend.
2. For Windows: evaluate WinFsp (Windows FUSE-compatible library) as an alternative to ProjFS.
3. Implement `VfsBackend` trait with `FuseBackend`, `WebDavBackend`, and `NfsBackend` implementations.
4. Allow users to select backend via configuration.
