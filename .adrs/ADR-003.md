# ADR-003: Virtual File System

## Status
Accepted

## Context
Suture needs to present repository contents as a mountable filesystem so non-linear editors (NLEs) can open and save files directly. The original spec called for NFSv4/SMB3 user-space servers for cross-platform loopback mounting, plus Windows ProjFS for native-speed virtualization. Each protocol has platform-specific advantages but also significant implementation complexity and platform restrictions.

## Decision
Use FUSE3 on Linux for full read/write filesystem mounting, and WebDAV as a cross-platform HTTP-based mount for macOS Finder, Windows Explorer, and Docker environments. The mount manager in `suture-daemon` handles FUSE/WebDAV lifecycle (mount, unmount, health checks).

## Consequences
### Positive
- FUSE3 on Linux provides full POSIX semantics: read, write, stat, mkdir, unlink, rename
- WebDAV is natively supported by macOS Finder (Cmd+K) and Windows Explorer (Map Network Drive) without additional software
- Single codebase for WebDAV server serves all non-Linux platforms
- FUSE3 integration is battle-tested (28 source files in `suture-vfs` crate)
- No kernel driver dependencies on macOS or Windows

### Negative
- FUSE3 requires `libfuse3-dev` and root for mount operations; integration tests are `#[ignore]`d in CI
- WebDAV lacks full POSIX semantics (no hard links, no advisory locks, limited rename atomicity)
- WebDAV performance is lower than native filesystem access due to HTTP overhead per operation
- ProjFS would provide native-speed file virtualization on Windows; WebDAV is slower for large directory listings

### Risks
- NLEs that require filesystem-level features (memory-mapped I/O, file locks) may not work over WebDAV. Mitigation: document limitations; FUSE3 on Linux provides full POSIX for supported environments.
- Windows users with large repos may find WebDAV too slow for interactive editing. Mitigation: evaluate WinFSP as a future FUSE equivalent for Windows.

## Alternatives Considered
| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|--------------|
| NFSv4 (nfs-serve-rs) | Native on macOS/Linux, good performance | No Windows support, complex NFSv4 state machine, requires rpcbind/portmapper | WebDAV covers macOS; NFSv4 adds platform-specific code for marginal gain |
| SMB3 (user-space) | Native Windows Explorer support, oplocks | Extremely complex protocol (SMB2/3 spec is ~1000 pages), limited Rust implementations | No mature Rust SMB3 server crate; WebDAV achieves the same Explorer integration |
| Windows ProjFS | Native-speed virtualization on Windows, kernel-level performance | Windows-only, requires Windows SDK and kernel driver signing, Rust bindings immature | Platform-specific investment not justified for current user base |
| WinFSP | Windows equivalent of FUSE, well-maintained | Windows-only, separate implementation from Linux FUSE | Could be added as a future Windows backend; WebDAV covers current needs |
