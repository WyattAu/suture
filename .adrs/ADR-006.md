# ADR-006: IPC Mechanism

## Status
Accepted

## Context
Suture daemon communicates with local CLI clients and editor plugins for status queries (is a file locked? what is the current branch? has sync completed?). The original spec called for iceoryx-rs for zero-copy shared-memory communication, targeting nanosecond-level IPC latency. iceoryx provides true zero-copy shared memory with a pub/sub model, lock-free queues, and bounded memory allocation. However, Suture's IPC pattern is low-frequency status queries, not high-throughput streaming, so nanosecond IPC is unnecessary.

## Decision
Use `memmap2` for memory-mapped shared memory segments between daemon and clients. The daemon writes status to a fixed-size SHM region; clients mmap the same region and read directly. Atomic primitives provide lock-free reads for progress tracking during large file pushes/pulls.

## Consequences
### Positive
- memmap2 is a lightweight crate (~800 LOC) with no unsafe beyond the mmap syscall wrapper
- Kernel-managed page caching means zero-copy reads after initial page fault
- Shared memory regions are file-backed, surviving daemon restarts if the backing file persists
- No daemon process required for reads: clients can mmap the SHM file independently
- Atomic operations on mmap'd memory provide lock-free progress tracking

### Negative
- No built-in pub/sub: clients must poll or use inotify on the SHM file for changes
- No message queue semantics: no backpressure, no message ordering guarantees
- Shared memory size must be fixed at creation; requires sizing estimates upfront
- No cross-machine IPC: memmap2 SHM is local to a single machine (which is the intended use case)

### Risks
- If a client crashes while holding a reference to the SHM region, stale data may persist. Mitigation: daemon writes a generation counter; clients detect stale data via counter mismatch.
- SHM file permissions must be set correctly to prevent unauthorized reads. Mitigation: create SHM files in a mode-0600 directory owned by the user.

## Alternatives Considered
| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|--------------|
| iceoryx-rs | True zero-copy pub/sub, nanosecond latency, lock-free, no allocator needed | Complex dependency chain (RouDi daemon), Rust bindings less mature, designed for automotive real-time systems | Over-engineered for low-frequency status queries |
| Unix domain sockets | Well-understood, bidirectional, no SHM management | Copy-based (not zero-copy), requires daemon to be running, higher latency (~10 us vs ~1 us for mmap) | Sufficient for control messages but unnecessary for bulk status data |
| D-Bus | Desktop-standard IPC, built-in signal mechanism | Copy-based, daemon dependency (dbus-daemon), complex protocol, not available in all environments | Heavyweight for Suture's simple status query pattern |
| Named pipes (FIFOs) | Simple, POSIX standard | Unidirectional, copy-based, no multiplexing | Too limited for bidirectional status exchange |
