# ADR-001: FlatBuffers Serialization

## Status
Accepted

## Context
Suture's wire protocol serializes patch-DAG nodes, OTIO timeline fragments, and binary project files. The original HFT-spec called for FlatBuffers to achieve zero-copy deserialization. Zero-copy is attractive when payloads are large and deserialization is on the hot path. However, patch payloads in typical video post-production workflows are small (median ~4 KB, P99 under 1 MB). The zero-copy benefit of FlatBuffers is marginal at these sizes compared to the ergonomics cost of its schema-driven codegen and restricted object model.

## Decision
Use `bincode` for serialization and `zstd` for compression. bincode is a serde-compatible format that integrates directly with Suture's existing Rust type definitions without a separate schema compilation step. Zstd delta-compression is applied on top for wire efficiency.

## Consequences
### Positive
- No build-time codegen step; bincode derives work with `#[derive(Serialize, Deserialize)]`
- Full serde ecosystem compatibility (JSON debug dumps, custom serializers)
- bincode serialization is ~2x faster than FlatBuffers for patch-sized payloads under 1 MB due to simpler encoding and no vtable indirection
- Zstd achieves 3-5x compression ratios on binary project files (OTIO XML, EDL)

### Negative
- Deserialization allocates; no zero-copy read path
- FlatBuffers would allow mmap-backed parsing of very large payloads without copying, which is lost
- bincode format is Rust-specific; cross-language clients need a translation layer

### Risks
- If payload sizes grow significantly (e.g., multi-GB timeline snapshots), deserialization cost may dominate. Mitigation: profile at scale and consider FlatBuffers or a hybrid approach for specific payload types.

## Alternatives Considered
| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|--------------|
| FlatBuffers | Zero-copy deserialization, cross-language schemas, deterministic layout | Requires `.fbs` schema files and codegen, restrictive object model (no Rust ownership), marginal benefit under 1 MB | Complexity not justified for current payload sizes |
| Protocol Buffers | Mature toolchain, cross-language, gRPC native | Slower than bincode for Rust-to-Rust, codegen required, proto3 semantics differ from Rust idioms | gRPC already used for transport; protobuf adds redundancy when all endpoints are Rust |
| Cap'n Proto | True zero-copy like FlatBuffers, Rust-native `capnp` crate | Schema compilation required, smaller ecosystem than serde | Same zero-copy rationale as FlatBuffers; ergonomics gap |
| MessagePack | Compact binary, cross-language | No zero-copy, slower than bincode in Rust benchmarks | No advantage over bincode for Rust-to-Rust communication |
