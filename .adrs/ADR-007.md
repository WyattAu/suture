# ADR-007: Python Bindings

## Status
Accepted

## Context
Suture provides Python bindings (`suture-py`) to allow DaVinci Resolve's Python scripting API and other Python-based tools to call `libsuture` directly. The original spec considered a C-ABI bridge approach, where a `cdylib` exports a flat C API that Python loads via `ctypes` or `cffi`. This approach is language-agnostic but introduces a manual FFI layer with no type safety. The alternative is PyO3, which generates native Python extension modules with automatic type conversion between Rust and Python.

## Decision
Use PyO3 with a `cdylib` crate to build a native Python extension module. PyO3 handles type marshalling, error conversion, and GIL management automatically. The `suture-py` crate exposes Python-callable functions for notes, worktree, blame, bisect, and remotes operations.

## Consequences
### Positive
- Native Python extension module: importable with `import suture` like any other Python package
- Automatic type conversion between Rust and Python types (strings, integers, lists, dicts)
- PyO3 manages the GIL correctly, allowing Rust code to release the GIL for CPU-bound work
- No manual C header files or ABI stability concerns
- maturin/PyO3 toolchain handles building, wheel packaging, and publishing to PyPI
- Type stubs (`.pyi`) can be auto-generated for IDE support

### Negative
- Tightly coupled to CPython ABI; each Python version (3.9, 3.10, 3.11, 3.12) requires a separate wheel
- `suture-py` is excluded from the Cargo workspace; CI does not test Python bindings (TD-5 in ROADMAP.md)
- PyO3 adds build complexity: requires Python dev headers, maturin, and platform-specific linking
- No cross-language ABI: PyO3 bindings cannot be reused by Node.js, C#, or other language runtimes

### Risks
- PyO3 version drift between `suture-py` and the main workspace. Mitigation: pin PyO3 version in `suture-py/Cargo.toml`; currently using PyO3 0.25.
- CPython ABI breakage on minor version updates (rare but possible). Mitigation: maturin builds version-specific wheels; users install the correct wheel for their Python version.

## Alternatives Considered
| Alternative | Pros | Cons | Why Rejected |
|-------------|------|------|--------------|
| C-ABI bridge (cdylib + ctypes/cffi) | Language-agnostic ABI, reusable by any FFI-capable language | Manual type marshalling, no GIL management, no error conversion, fragile ABI stability | Unnecessary indirection when the only consumer is Python |
| PyO3 + c-ABI hybrid | PyO3 for Python, exported C functions for other languages | Dual maintenance, complexity | No current demand for non-Python bindings beyond Node.js (which uses napi-rs) |
| JSON-RPC over HTTP | Language-agnostic, no native extensions needed | Network latency, no direct memory access, serialization overhead | Defeats the purpose of "direct without IPC overhead" |
| subprocess CLI wrapper | Simplest integration, no native extensions | Highest latency, no streaming, brittle output parsing | Unacceptable for editor integration where responsiveness matters |
