# Acceptance Criteria: Suture — Universal Semantic Version Control System

**Document ID:** SPEC-AC-001
**Status:** Draft
**Date:** 2026-03-27

---

## Priority Definitions

| Priority | Definition |
|:---|:---|
| **P0 — Critical** | Must pass before any release. Blocks ship. |
| **P1 — High** | Must pass before GA. Required for target market. |
| **P2 — Medium** | Should pass. Tracked for future milestone. |

## Core Engine (REQ-CORE)

| Req ID | Acceptance Criterion | Verification Method | Priority |
|:---|:---|:---|:---|
| REQ-CORE-001 | The compiled `libsuture` crate exposes public APIs for patch algebra, CAS, and DAG operations. | `cargo doc --no-deps` succeeds; public API surface matches architectural spec. | P0 |
| REQ-CORE-002 | Given the same set of patches in any valid commutative order, the resulting project state hash is identical across 100 runs. | Property-based test (`proptest`) with randomized patch orderings; assert state hash equality. | P0 |
| REQ-CORE-003 | Replaying a patch set of 1,000 patches produces the same DAG state as the initial application. | Integration test: apply patches, serialize DAG, deserialize, replay, compare hash. | P0 |
| REQ-CORE-004 | 8 concurrent readers can traverse the DAG without deadlocks over a 10-second stress test. | Concurrency test: 8 tokio tasks reading DAG simultaneously; assert no deadlock within timeout. | P0 |
| REQ-CORE-005 | Under concurrent writes (4 writers), the DAG never enters an inconsistent state. | Stress test: 4 tokio tasks committing patches; post-test DAG integrity check passes. | P0 |
| REQ-CORE-006 | `libsuture` compiles as a library crate with no default features required. | `cargo build -p suture-core --no-default-features` succeeds. | P1 |
| REQ-CORE-007 | All I/O-bound operations in the core engine are async and return futures compatible with tokio. | Static analysis: no blocking I/O calls outside of `spawn_blocking` wrappers. | P1 |

## Content Addressable Storage (REQ-CAS)

| Req ID | Acceptance Criterion | Verification Method | Priority |
|:---|:---|:---|:---|
| REQ-CAS-001 | Storing and retrieving a 1 GB file produces the same BLAKE3 hash as the original. | Integration test: write 1 GB random data, read back, compare BLAKE3 hashes. | P0 |
| REQ-CAS-002 | Stored blobs occupy less disk space than the original input (compression ratio > 1.0x for compressible data). | Benchmark: compress 1 GB of mixed data (text, JSON, XML); assert compressed size < original. | P0 |
| REQ-CAS-003 | Writing the same blob 100 times results in exactly 1 physical copy on disk. | Test: write identical 10 MB blob 100 times; count files in CAS directory; assert count == 1. | P0 |
| REQ-CAS-004 | A virtual blob referencing an external 10 GB video file occupies less than 1 KB in the CAS. | Test: create virtual blob for external file; assert CAS entry size < 1 KB; assert external file is not copied. | P1 |
| REQ-CAS-005 | For 1,000 randomly generated byte sequences (1 KB to 100 MB), decompression produces byte-identical output. | Property-based test: generate, compress, decompress, compare with `assert_eq!`. | P0 |
| REQ-CAS-006 | BLAKE3 hashing throughput exceeds 1 GB/s single-thread on x86_64 with AVX2. | Benchmark (`criterion`): hash 1 GB buffer; assert throughput > 1 GB/s. | P0 |
| REQ-CAS-007 | CAS write throughput exceeds 100 MB/s with Zstd level 3. | Benchmark (`criterion`): write 1 GB of compressible data; assert throughput > 100 MB/s. | P0 |
| REQ-CAS-008 | A repository initialized with `--hash-algorithm sha3-256` uses SHA3-256 for all CAS operations. | Integration test: init with SHA3-256, write blob, verify hash prefix matches SHA3-256. | P2 |
| REQ-CAS-009 | After creating a branch, committing to it, and deleting the branch, `gc` reclaims blobs only reachable from the deleted branch. | Integration test: branch, commit, delete branch, run gc; assert unreferenced blob count > 0 and disk usage decreases. | P1 |
| REQ-CAS-010 | Killing the process mid-write does not leave a partial blob readable from the CAS. | Fault injection test: write 1 GB blob, kill process at random offset, restart, attempt read; assert error or clean state. | P0 |

## Patch Algebra (REQ-PATCH)

| Req ID | Acceptance Criterion | Verification Method | Priority |
|:---|:---|:---|:---|
| REQ-PATCH-001 | A patch of type `UpdateNode` with touch set `{track_1, clip_a}` serializes to a valid FlatBuffers buffer and deserializes losslessly. | Unit test: serialize, deserialize, compare fields. | P0 |
| REQ-PATCH-002 | Two patches with disjoint touch sets commute — `apply(A, apply(B, state)) == apply(B, apply(A, state))`. | Property-based test: generate random disjoint patches, apply in both orders, assert state equality. | P0 |
| REQ-PATCH-003 | Merging two branches with 500 commuting patches each produces a merged state equal to the set-union of all patches. | Integration test: create two branches, apply patches, merge, compare merged state with expected union. | P0 |
| REQ-PATCH-004 | Merging two branches with overlapping touch sets produces a conflict node containing both patch sets intact. | Integration test: create conflicting patches, merge, assert conflict node exists with both sides preserved. | P0 |
| REQ-PATCH-005 | A conflict node contains complete data from both branches — no field is null, truncated, or lost. | Unit test: inspect conflict node fields; assert both `ours` and `theirs` are fully populated. | P0 |
| REQ-PATCH-006 | Applying the identity patch to any state leaves the state unchanged. | Property-based test: generate random states, apply identity, assert equality. | P0 |
| REQ-PATCH-007 | For any three patches P1, P2, P3: `apply(P3, apply(P2, apply(P1, state))) == apply(P1, apply(P3, apply(P2, state)))` (when all commute). | Property-based test: random commuting patch triples, both orderings, assert equality. | P0 |
| REQ-PATCH-008 | The `proptest` suite for commutativity, associativity, and identity runs without failure over 10,000 iterations. | CI test: `cargo test -p suture-core patch_algebra_proptest`; assert 0 failures. | P1 |
| REQ-PATCH-009 | Patches round-trip through FlatBuffers serialization/deserialization without data loss. | Unit test: serialize patch to FlatBuffers bytes, deserialize, compare all fields. | P0 |
| REQ-PATCH-010 | Patch metadata (touch set, type, parents) is stored as FlatBuffers and retrievable from the metadata database. | Integration test: commit patch, query metadata DB, assert FlatBuffers fields match. | P0 |
